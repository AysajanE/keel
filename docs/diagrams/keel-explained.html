Keel

The idea → design → plan → execute → ship toolchain. Daily commands live in ~/keel/bin/keel-*.

Four tools that take you from "I have an idea" to "I shipped a working product." Built from first principles, on one rule: every tool talks through one shared contract — never by importing each other's code.

1. The big picture

You start with an idea. You end with a shipped product. The four tools are stops on that journey. Click any box to highlight its role; click again to clear.

One sentence: gstack helps you decide what to build, the compiler turns that into a precise recipe, plan-orchestrator builds it one step at a time with quality checks, and gstack helps you ship the result. SWR is an optional safer-but-slower replacement for the compiler when the stakes are high.

2. First principles — why four tools instead of one

You could imagine asking one giant AI agent: "Build me a health data hub." What goes wrong?

⚠ Problem A: Conversation drifts

The AI has good ideas, asks good questions — and then forgets what you decided three turns ago. The plan grows fuzzy. You can't tell what's "decided" vs. "still talking."

Solution: separate thinking from doing. Lock decisions in a written design doc before any code is touched.

⚠ Problem B: Execution is opaque

The AI writes code. Does it work? Does it pass tests? Did it touch files it shouldn't have? Without strict process you find out only when something breaks in production.

Solution: make execution auditable. One step at a time, in an isolated git worktree, with two independent reviewers (Codex + Claude) inspecting the same evidence.

The architectural insight

Don't build one big tool that does everything. Build small tools, each excellent at one job, and have them talk through one shared file format:

  • gstack excels at human conversation and review.
  • compiler excels at deterministic translation: design doc → precise recipe.
  • plan-orchestrator excels at strict, audited execution.
  • SWR excels at multi-agent reviewed planning (only when needed).

The shared format is markdown_playbook_v1. Everything between "design doc" and "running code" passes through it. That's the whole trick: a clean handoff between humans (who think) and the executor (which runs).

Why this matters in practice: If you ever want to replace the compiler with something better, you can — as long as the new tool still emits valid markdown_playbook_v1. The executor never changes. Same goes the other way: a smarter executor can drop in later. Contracts make systems evolvable.

3. Each tool, in plain English

Click a tab. Each shows what the tool is, what it does, what it deliberately refuses to do, and how you use it.

cockpit   Built by Garry Tan (President of YC). Installed at ~/.claude/skills/gstack/. 23 slash commands that act as a virtual product team inside Claude Code.

What it is

A library of Markdown skills. Each skill is one specialist:

  • /office-hours — six forcing questions to lock down what you're building
  • /autoplan — runs CEO, design, eng, DX reviews automatically
  • /plan-eng-review — locks architecture and tests
  • /review — finds production bugs before they ship
  • /qa — opens a real browser, finds bugs, fixes with regression tests
  • /cso — security audit (OWASP + STRIDE)
  • /ship — opens a PR with version, CHANGELOG, doc sync
  • /land-and-deploy — merges, deploys, verifies
  • /retro, /learn — weekly retrospective + persistent memory

What it doesn't do

  • It does not execute structured code-change plans. That's plan-orchestrator's job.
  • It does not own state. ~/.gstack/projects/ is working memory; authoritative artifacts get promoted into the repo.
  • It does not approve plan-orchestrator's manual gates. Only you do, from your terminal.

How you use it

Inside Claude Code. Just type /office-hours and describe what you're building. The skill takes over.

Mental model: gstack is the front and back of the loop. It helps you decide what to build (front) and helps you ship the result (back). In between, the compiler and plan-orchestrator do the structured work.

translator   Lives at ~/keel/tools/compiler. Invoke via keel-compile. It bridges the gap between a gstack design doc (free-form conversation) and what plan-orchestrator needs (a strict structured recipe). Current stub output is scaffold-only until a real LLM row author is implemented.

What it is

A small Python program with four stages:

1. Python parses your gstack design doc
   → gstack_plan_ir_v1.json  (deterministic facts)

2. Candidate row author
   → po_candidate_rows_v1.json  (JSON rows only)
   current shipped author: stub only

3. Python compiler preflight
   → compiler_validation_report_v1.json
   (fails closed on bad paths, broad write-roots,
    missing red/green commands, etc.)

4. Python emits the final markdown
   → docs/playbooks/<slug>.playbook.md

5. PO validates its own contract
   → list-items + doctor --playbook

The key rule

A future LLM author may write candidate rows. It never writes the final Markdown table. Python owns:

  • Markdown table formatting (avoids parser drift)
  • Validation verdicts (fails closed)
  • Narrow-roots safety checks
  • Running PO contract verification when the PO root is known

What it doesn't do

  • Never launches plan-orchestrator execution.
  • Never invents file paths the parser didn't observe in your design doc.
  • Never calls mark-manual-gate. Only you do.
  • Doesn't do multi-agent staged review. That's SWR's job.
v0 status (today): The compiler runs end-to-end with a deterministic stub author (no LLM API needed), but stub output is scaffold-only. Non-dry-run stub output fails unless --allow-stub-output is explicit, and PO list-items + doctor is mandatory when --plan-orchestrator-root is provided. The real LLM authors still raise NotImplementedError.

executor   Lives at plan-orchestrator/. The strict, careful runtime that turns a reviewed playbook into actual code changes — one step at a time.

What it is

A state machine. Each row of your playbook becomes one item that grinds through these states:

setup_worktree → execute → verify
  → audit_packet_ready
  → audit_codex AND audit_claude (same packet)
  → triage_merge (deterministic)
  → one of four terminals:
       passed
       awaiting_human_gate
       blocked_external
       escalated

Each item attempt lives in its own git worktree branch (orchestrator/item/<run_id>/<item_id>/attempt-N). Bad code can never leak into your main branch.

What it doesn't do

  • Doesn't plan. Doesn't invent items. Takes one playbook file as input, that's it.
  • Doesn't browse the web.
  • Doesn't auto-approve human gates.
  • Doesn't fabricate external evidence. If a step needs human-supplied evidence, it stops at blocked_external and waits.

The dual-audit trick

After verification, Codex and Claude both review the same frozen packet. Findings are canonicalized by hash. Two independent witnesses, not one.

The supervisor layer (supervise run) is an additive wrapper that adds heartbeats, fresh liveness probes, and bounded recovery — but the kernel state in run_state.json remains authoritative. The supervisor never approves a manual gate. Ever.

optional rigor   Lives at staged-workflow-runner/. A more careful, slower target path for high-stakes playbook generation, with multi-agent staged review along the way.

What it is

A manifest-driven runner for staged OpenAI Responses workflows. The generic gstack-to-PO task pack is a scaffold target; it should be used only after PO contract validation and human review:

Stage 1: Scope and source authority
Stage 2: Repo grounding and surfaces
Stage 3: Execution row drafting
Stage 4: Gates, risks, verification
Stage 5: Final playbook emission
        → markdown_playbook_v1
          (same shape the compiler produces)

Between every stage, three agents review independently: operator Codex, read-only Codex, read-only Claude. You explicitly accept only changes that have applied-change evidence.

When to use it instead of the compiler

The human picks — it's never auto-forced. Suggested triggers:

  • security / auth / secrets / payments / signing
  • data migration or any irreversible write
  • regulated or customer-trust-facing claims
  • large multi-stage feature with ambiguous ordering
  • broad repo blast radius
  • you simply want multi-agent staged review before code touches the repo

What it doesn't do

  • Doesn't execute. The terminal stage emits a playbook; plan-orchestrator runs it.
  • Doesn't replace plan-orchestrator. They handle different stages of the same pipeline.
For a small internal tool: SWR is probably not needed. If the scope is personal-use, low-risk, and easy to review, use the fast lane and reserve SWR for workflows involving money, customers, durable state, or broad blast radius.

4. The contract that holds it together

Everything in between humans and the executor passes through one file format: markdown_playbook_v1.

What a markdown_playbook_v1 looks like

compiled_by: gstack_to_markdown_playbook_v1
compiled_at: 2026-05-15T21:10:47+00:00
human_approved_by: operator
source_artifacts:
  - kind: office_hours, path: docs/gstack/my-feature-office-hours.md, sha256: ...
-->

5. Two paths through the engine

⚡ Fast lane scaffold

Use this for low-risk planning only after treating compiler output as a scaffold that needs human review.

1. /office-hours (gstack)
   → design doc

2. /autoplan (gstack)
   → reviewed plan

3. compiler
   → markdown_playbook_v1 scaffold
   → PO list-items + doctor verification

4. human review
   → accept, revise, or discard scaffold

5. plan-orchestrator
   → completed branch

6. /ship + /land-and-deploy (gstack)
   → live in production

The deterministic compiler author is scaffold-only. Non-dry-run stub output is blocked unless the operator explicitly passes --allow-stub-output; real LLM authors are reserved for future implementation.

🛡 High-stakes lane

Use this when the human owner judges the stakes high: security, money, irreversible writes, compliance, or customer-facing claims.

1. /office-hours (gstack)
   → design doc

2. /autoplan (gstack)
   → reviewed plan
   → approved build brief

3. SWR gstack_design_to_po_playbook pack
   → staged, reviewed markdown_playbook_v1 draft
   → PO list-items + doctor verification

4. human review
   → accept, revise, or discard playbook

5. plan-orchestrator
   → completed branch

6. /ship + /land-and-deploy (gstack)

Slower and more expensive. Worth it when the cost of a bad plan is higher than the cost of a careful review process.

Both lanes target the same contract, not the same trust level. plan-orchestrator only executes a file that validates as markdown_playbook_v1, and the compiler now runs PO list-items plus doctor when a PO root is known. Human review remains mandatory before execution.

6. Walkthrough: a feature, end-to-end

Concrete example using a promoted gstack design doc. Click each phase to expand commands and artifacts.

phase 0 Idea → design doc

In Claude Code, you typed:

/office-hours

You said: "I want to build a small feature that turns a reviewed product idea into a concrete implementation plan."

Six rounds of forcing questions later, you had a design doc:

~/.gstack/projects/my-product/
  local-my-product-feature-design-YYYYMMDD-HHMMSS.md

Status: approved after human review.

Human gate at the end of this phase:

  • You confirm the design doc reflects what you actually want to build.
phase 1 Design → reviewed plan → approved brief

Promote the design doc into the target repo, then review it:

# Copy the gstack design doc into the product repo
export PRODUCT_REPO="$HOME/workspaces/my-product"
cp ~/.gstack/projects/my-product/local-my-product-feature-design-YYYYMMDD-HHMMSS.md \
   "$PRODUCT_REPO/docs/gstack/my-feature-office-hours.md"

# In Claude Code, run the review gauntlet
/autoplan docs/gstack/my-feature-office-hours.md

This walks through CEO review → design review → eng review → DX review, with you making decisions at each step. Output:

docs/gstack/my-feature-autoplan.md
docs/briefs/my-feature.approved-brief.md

Human gate at the end of this phase:

  • You confirm the approved build brief is what you want to execute.
phase 2A Brief → playbook scaffold (fast lane)

Run the compiler (uses the keel-compile wrapper from ~/keel/bin/):

source ~/keel/keel.env
export PRODUCT_REPO="$HOME/workspaces/my-product"

keel-compile compile \
  --repo-root "$PRODUCT_REPO" \
  --design   "$PRODUCT_REPO/docs/gstack/my-feature-office-hours.md" \
  --autoplan "$PRODUCT_REPO/docs/gstack/my-feature-autoplan.md" \
  --approved-brief "$PRODUCT_REPO/docs/briefs/my-feature.approved-brief.md" \
  --out      "$PRODUCT_REPO/docs/playbooks/my-feature.playbook.md" \
  --plan-orchestrator-root "$KEEL_PO_ROOT" \
  --human-approved-by "$USER" \
  --dry-run

Current alpha behavior is intentionally conservative: the deterministic row author is scaffold-only. To write stub output anyway, an operator must pass --allow-stub-output, and the compiler forces generated rows through signoff gates.

Outputs three artifacts:

docs/playbooks/my-feature.playbook.md          (candidate playbook scaffold)
docs/playbooks/my-feature.playbook.meta.json   (compiler provenance)
docs/playbooks/my-feature.validation.json      (validation report)

The compiler performs the PO contract check when --plan-orchestrator-root is supplied. If the PO root is intentionally unavailable, the operator must pass --skip-po-verify "<reason>" and record why the contract check was deferred.

Human gate at the end of this phase:

  • You read the playbook end-to-end and confirm it's what you want plan-orchestrator to execute.
  • Validation status is pass and PO contract verification passed or has a recorded skip reason.
phase 2B Brief → playbook (high-stakes lane, optional)

For low-risk work, skip this and use the compiler scaffold path, then review before PO execution.

If you did need it for a higher-risk feature:

# 1. Initialize an SWR session (keel-swr supervisor → run_responses_supervisor_v2.py)
keel-swr supervisor init-session \
  --root "$PRODUCT_REPO" \
  --clarified-task-brief docs/briefs/<slug>.approved-brief.md \
  --summary "Emit a reviewed markdown_playbook_v1 for <slug> v1."

# 2. Stage + examine + dry-run the playbook-emission scaffold
# 3. Run the 5 staged calls (each with multi-agent review between)
# 4. Promote terminal artifact
cp <SWR_RUN_DIR>/stages/05_final_markdown_playbook/response.final.md \
   docs/playbooks/<slug>.playbook.md

# 5. Validate with keel-run list-items + keel-doctor --playbook (same as the fast lane)
phase 3 Playbook → executed code, one item at a time

Run plan-orchestrator under supervision (uses keel-run):

source ~/keel/keel.env
cd "$PRODUCT_REPO"

# Acknowledge any ambient agent configs once after reviewing them
export PLAN_ORCHESTRATOR_CLEAN_ENV_CONFIRMED=1

keel-run supervise run \
  --playbook docs/playbooks/my-feature.playbook.md \
  --next

For each item, plan-orchestrator:

  1. Creates a fresh git worktree: .local/automation/plan_orchestrator/worktrees/<RUN_ID>/item-01-attempt-1/
  2. Executes the item (Codex implements the action)
  3. Runs the verification command(s) inside the worktree
  4. Freezes an audit packet (patch + verification report + snapshots)
  5. Asks Codex to audit the frozen packet
  6. Asks Claude to audit the same frozen packet
  7. Deterministically merges findings, hashing duplicates
  8. Triages: passed / awaiting_human_gate / blocked_external / escalated

If the item lands passed, plan-orchestrator fast-forwards the run branch orchestrator/run/<RUN_ID> and moves to the next item.

You can watch live:

keel-run supervise status --run-id <RUN_ID> --format json --exit-code
phase 4 Manual gates — you, in a human-held terminal

When plan-orchestrator reaches awaiting_human_gate, it stops. You read:

  • manual_gate.json
  • candidate patch
  • verification report
  • Codex audit report
  • Claude audit report
  • triage report

Then in your own terminal (no agent), you record the decision:

keel-run mark-manual-gate \
  --run-id <RUN_ID> \
  --item 02 \
  --decision approved \
  --by "$USER" \
  --note "Reviewed the gate packet and approve continuation." \
  --evidence-path docs/reviews/my-feature-02-signoff.md \
  --approval-token-file /secure/local/path/manual-gate-token.txt

Then resume:

keel-run supervise resume --run-id <RUN_ID>
Critical rule: no agent, no wrapper, no skill, no compiler ever calls mark-manual-gate. You do. This is the single unbreakable boundary of the system.
phase 5 External evidence — when the item needs proof from outside the repo

Some items need evidence plan-orchestrator can't fetch on its own (e.g., "verified the Oura webhook works for my actual account"). When it reaches blocked_external, you supply local evidence:

# Put proof files in a local directory
mkdir -p /tmp/oura-evidence
# (drop screenshots, logs, etc.)

python .../run_plan_orchestrator.py supervise resume \
  --run-id <RUN_ID> \
  --external-evidence-dir /tmp/oura-evidence

plan-orchestrator copies the directory into the item's evidence folder and resumes.

External evidence is local-file only. No agent fabricates it, no script downloads it. You hand it over physically.
phase 6 Passed branch → PR → deploy → retrospective

When all items reach passed, plan-orchestrator's work is done. The completed code lives on branch orchestrator/run/<RUN_ID>.

Hand off to gstack — but never directly:

# 1. Create a normal ship branch from the orchestrator run branch
git fetch origin
git checkout -b ship/my-feature orchestrator/run/<RUN_ID>

# 2. Let gstack prepare the PR (tests, review, version bump, CHANGELOG)
/ship

# 3. Let gstack merge and deploy after the PR exists
/land-and-deploy

# 4. (For web apps) post-deploy monitoring
/canary

# 5. Documentation sync
/document-release

# 6. Capture learnings
/retro
/learn
Why a separate ship branch? orchestrator/run/<RUN_ID> is internal kernel state. gstack /ship is a release workflow that does its own merging, version bumping, and PR opening. Keeping them separate means you never confuse "the executor's working state" with "the thing we're proposing to ship."

7. What happens after plan-orchestrator finishes

plan-orchestrator's last act is to report one of four terminal states. The next move is yours — but the playbook tells you what to do.

✓ passed

All items completed cleanly. orchestrator/run/<RUN_ID> is fast-forwarded with the work.

Next: git checkout -b ship/<slug> orchestrator/run/<RUN_ID>, then gstack /ship.

⏸ awaiting_human_gate

An item declared manual_gate and is waiting for your signoff.

Next: read the gate packet, then mark-manual-gate from your terminal, then supervise resume.

⏸ blocked_external

An item needs evidence that lives outside the repo.

Next: assemble the evidence directory, then supervise resume --external-evidence-dir.

⚠ escalated

Critical unresolved issue, infrastructure failure, or rejected manual gate. No automatic recovery path.

Next: read the audit reports and triage report, decide whether to revise the playbook, fix manually, or abandon the run.

The system is designed so "stop and ask" is always available. No matter where plan-orchestrator ends up, you can status, doctor, or inspect the worktree manually. Nothing is ever silently abandoned.

8. "Which tool should I use?" decision guide

Q: I have an idea. Where do I start?

→ gstack /office-hours

Always. Even for tiny features. The forcing questions surface what you don't know yet.

Q: I have an approved design. How do I get a playbook?

Is the work high-stakes (security, money, irreversible state, broad blast radius)?

No (most cases) → compiler (this repo). Fast, deterministic.
Yes → SWR. Five staged calls, multi-agent reviewed between each.

Q: My playbook is ready. How do I execute it?

→ plan-orchestrator supervise run --next

Always. The compiler and SWR never execute. plan-orchestrator is the only executor.

Q: plan-orchestrator stopped at awaiting_human_gate. What now?

Open a fresh terminal that no agent controls. Read the gate packet. If you approve:

mark-manual-gate --decision approved --by "<you>" ... --approval-token-file ...
supervise resume --run-id <RUN_ID>

Q: plan-orchestrator finished with passed. How do I ship?

→ git checkout -b ship/<slug> orchestrator/run/<RUN_ID> → gstack /ship → gstack /land-and-deploy

Always create a sibling ship branch. /ship never operates directly on the kernel's run branch.

9. Safety boundaries that must not be crossed

The system has carefully separated authorities. Crossing any of these lines breaks the safety properties that make the whole thing trustworthy.

🚫 No agent approves manual gates

Only you, from a human-held terminal, can call mark-manual-gate. Compilers, supervisors, skills, wrappers — none of them. Optionally protect this further with the env var PLAN_ORCHESTRATOR_MANUAL_GATE_TOKEN_SHA256.

🚫 No fabricated external evidence

When plan-orchestrator says blocked_external, evidence must come from a local directory you physically supply. No agent invents proof.

🚫 No "." or broad write roots

allowed_write_roots must be narrow (1–3 entries, repo-relative). Never ., src, tests. Never .local, .git, .codex, .claude, .mcp.json, secrets, .env*, ops/config.

🚫 No skipping verification

Behavioral items (requires_red_green: true) must have at least one required_verification_commands entry that would fail without the change. No exceptions.

🚫 No merging supervisors

plan-orchestrator's supervisor and SWR's supervisor stay separate. Merging them adds failure surface without improving anything.

🚫 No shipping the run branch directly

orchestrator/run/<RUN_ID> is internal state. Always create a sibling ship/<slug> branch and let gstack /ship drive the PR.
