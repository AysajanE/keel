Keel

The local-first control plane that takes a reviewed idea, compiles it into a strict execution recipe, runs it under audit, and hands the result back to you to ship.

Keel is built on one premise: the safest way to turn an idea into shipped code is to keep thinking, translating, executing, and shipping in separate tools — each excellent at one job — and have them talk only through written contracts that humans can read and approve. This guide rebuilds that premise from the ground up using interactive diagrams.

2. First principles — why split things up

You could imagine asking one giant AI agent: "Build me a health data hub." What goes wrong?

⚠ Problem A — Conversation drifts

The AI has good ideas, asks good questions, and then forgets what you decided three turns ago. The plan grows fuzzy. You cannot tell what is decided versus what is still being talked about.

Fix: separate thinking from doing. Lock decisions in a written design doc before any code is touched.

⚠ Problem B — Execution is opaque

The AI writes code. Does it work? Does it pass tests? Did it touch files it should not have? Without strict process you find out only when something breaks.

Fix: make execution auditable. One item at a time, in an isolated git worktree, with two independent reviewers inspecting the same evidence.

⚠ Problem C — Tools become entangled

If gstack imports plan-orchestrator and plan-orchestrator imports SWR, you cannot replace any one tool without breaking the others. Improvements stall.

Fix: every tool talks to its neighbors through a contract, never by importing each other's code. Contracts make systems evolvable.

⚠ Problem D — Installs are fragile

If the wrappers, the tools, and the user's product code all live in one repo, every change to one tool ripples through everything.

Fix: make Keel a meta-repo. Wrappers and contracts live here; the tools are cloned in at install time from their own canonical repos.

The architectural insight

Don't build one big tool that does everything. Build small tools, each excellent at one job, and have them talk through one shared file format and one declarative manifest. The result is four core tools, two contracts, one wrapping repository — plus one optional integration (gbrain) for users who want a shared memory layer.

3. The two contracts that hold Keel together

Keel is built on two contracts living at two different layers. Both must hold for the system to stay safe and replaceable.

Click either contract above to see what it guarantees, what fails closed, and why it exists at its specific layer.

Why two contracts, not one?

The install-time contract answers "which tool versions does this machine trust?" The runtime contract answers "is this playbook safe to execute?" Those are different questions on different timelines — you change tool versions occasionally, you write new playbooks constantly. Splitting them means each kind of change touches only one contract, so each review stays small.

What replaceability buys you

If a better compiler appears tomorrow, you can swap it in — as long as the new tool still emits valid markdown_playbook_v1 and still passes the manifest health check. Plan-orchestrator never changes. The contract is the bridge.

4. Meta-repo anatomy: what lives where

Keel is a meta-repo. It does not ship the tools' source code. It ships only what you need to coordinate them: wrappers, a manifest, an installer, examples, and documentation. The tools themselves are cloned in at install time.

Click each layer to see what lives there, what is tracked, and why the boundary exists.
The single rule of thumb: if a file changes per-user or per-install, it does not belong in the Keel repo. The Keel repo is the same on every machine. tools/, keel.env, tools.lock, and private/ are all generated locally and gitignored.

5. How install.sh bootstraps the system

Installing Keel is two human commands. Behind those commands, the installer walks a fixed pipeline. Click any step to see what it does.

Click any step to see what it actually does, what fails closed, and what flags affect it.
# Two-command bootstrap
git clone https://github.com/AysajanE/keel.git ~/keel
cd ~/keel
./install.sh                       # full install — clones manifest tools
source ~/keel/keel.env             # exports KEEL_ROOT, KEEL_PO_ROOT, etc., adds bin/ to PATH

# Optional: add gbrain
./install.sh --with gbrain

# Variants
./install.sh --public-only         # skip private tools (compiler still private)
./install.sh --skip-tools          # only write env/lock; do not clone
./install.sh --update-tools        # fetch manifest refs into clean checkouts
./install.sh --check               # validate manifest + scripts only
./install.sh --release-gate        # enforce: tag refs, no blocker statuses, required tools public
Why a manifest, not a script? Because the manifest is a declarative trust boundary. scripts/manifest.py validates every field against strict allowlists — health-check commands must match a regex allowlist, refs must be commit/tag/branch, URLs must be HTTPS GitHub, and so on. Adding a tool is a reviewable change with a tiny blast radius.

6. The four core tools (plus one optional integration)

Click a tab. The first four are the core idea-to-ship loop. The fifth tab (gbrain) is an optional third-party memory layer. Each shows what the tool is, what it does, what it deliberately refuses to do, and how you use it.

cockpit   Third-party. Built by Garry Tan (President of YC). Installed at ~/.claude/skills/gstack/. A library of 23 Markdown slash-command skills that act as a virtual product team inside Claude Code. Not part of Keel, but Keel is designed around it.

What it is

One library, many specialists. Each skill is one role:

  • /office-hours — six forcing questions to lock down what you're building
  • /autoplan — runs CEO, design, eng, DX reviews automatically
  • /plan-eng-review — locks architecture and tests
  • /review — finds production bugs before they ship
  • /qa — opens a real browser, finds bugs, fixes with regression tests
  • /cso — security audit (OWASP + STRIDE)
  • /ship — opens a PR with version, CHANGELOG, doc sync
  • /land-and-deploy — merges, deploys, verifies
  • /retro, /learn — weekly retrospective + persistent memory

What it doesn't do

  • It does not execute structured code-change plans. That is plan-orchestrator's job.
  • It does not own state. ~/.gstack/projects/ is working memory; authoritative artifacts must be promoted into the product repo.
  • It does not approve plan-orchestrator's manual gates. Only you do, from your terminal.

How you use it

Inside Claude Code. Just type /office-hours and describe what you're building. The skill takes over. Keel does not wrap gstack — it depends on the artifacts gstack produces.

Mental model: gstack is the front and back of the loop. It helps you decide what to build (front) and helps you ship the result (back). In between, the compiler (or SWR) and plan-orchestrator do the structured work.

translator   Lives at ~/keel/tools/compiler/ (cloned from AysajanE/gstack-playbook-compiler, currently private). Invoked via keel-compile. It bridges the gap between a gstack design doc (free-form conversation) and what plan-orchestrator needs (a strict structured recipe).

What it is

A small Python program with four stages and one mandatory post-check:

1. Python parses gstack artifacts
   → gstack_plan_ir_v1.json
     (deterministic facts only)

2. Candidate row author
   → po_candidate_rows_v1.json
   Currently shipped: stub author only.
   Future: per-phase LLM author.

3. Python preflight validation
   → compiler_validation_report_v1.json
   Fails closed on broad write-roots,
   missing red/green commands, etc.

4. Python emits the final markdown
   → docs/playbooks/<slug>.playbook.md

5. PO contract verification
   list-items + doctor --playbook
   (runs when --plan-orchestrator-root is given)

The key rule

An LLM author may write candidate rows in the future. It never writes the final Markdown table. Python always owns:

  • Markdown table formatting (avoids parser drift)
  • Validation verdicts (fails closed)
  • Narrow-write-roots safety checks
  • PO contract verification when a PO root is known

What it doesn't do

  • Never launches plan-orchestrator execution.
  • Never invents file paths the parser did not observe.
  • Never calls mark-manual-gate. Only you do.
  • Does not do multi-agent staged review. That is SWR's job.
Alpha status today: the compiler runs end-to-end but its deterministic stub author is scaffold-only. Non-dry-run stub output is blocked unless an operator explicitly passes --allow-stub-output, and PO list-items + doctor is required when --plan-orchestrator-root is provided. The real LLM authors raise NotImplementedError until they ship.

executor   Lives at ~/keel/tools/plan-orchestrator/ (cloned from AysajanE/plan-orchestrator, public). Invoked via keel-run. The strict, careful runtime that turns a reviewed playbook into actual code changes — one item at a time.

What it is

A state machine. Each row of your playbook becomes one item that walks through these states:

setup_worktree → execute → verify
  → audit_packet_ready
  → audit_codex AND audit_claude
     (same frozen packet)
  → triaging (deterministic merge)
  → one of four terminals:
       passed
       awaiting_human_gate
       blocked_external
       escalated

Each item attempt lives in its own git worktree branch orchestrator/item/<run_id>/<item_id>/attempt-N. Bad code can never leak into your main branch.

What it doesn't do

  • Does not plan. Does not invent items. Takes one playbook file as input, that is it.
  • Does not browse the web.
  • Does not auto-approve human gates.
  • Does not fabricate external evidence. If a step needs human-supplied evidence, it stops at blocked_external and waits.

The dual-audit trick

After verification, Codex and Claude both review the same frozen packet. Findings are canonicalized by hash and merged deterministically. Two independent witnesses, not one.

The supervisor layer (keel-run supervise run) is an additive wrapper that adds heartbeats, fresh liveness probes, and bounded recovery — but the kernel state in run_state.json remains authoritative. The supervisor never approves a manual gate. Ever.

optional rigor   Lives at ~/keel/tools/staged-workflow-runner/ (cloned from AysajanE/staged-workflow-runner, public). Invoked via keel-swr. A more careful, slower target path for high-stakes playbook generation, with multi-agent staged review along the way.

What it is

A manifest-driven runner for staged OpenAI Responses workflows. The generic gstack_design_to_po_playbook task pack emits a reviewed playbook in five stages:

Stage 1: Scope and source authority
Stage 2: Repo grounding and surfaces
Stage 3: Execution row drafting
Stage 4: Gates, risks, verification
Stage 5: Final playbook emission
        → markdown_playbook_v1
          (same contract the compiler produces)

Between every stage, three agents review independently: operator Codex, read-only Codex, read-only Claude. You explicitly accept only changes that have applied-change evidence.

When to use it instead of the compiler

The human picks — it is never auto-forced. Suggested triggers:

  • security · auth · secrets · payments · signing
  • data migration or any irreversible write
  • regulated or customer-trust-facing claims
  • large multi-stage feature with ambiguous ordering
  • broad repo blast radius
  • you simply want multi-agent staged review before code touches the repo

What it doesn't do

  • Does not execute. The terminal stage emits a playbook; plan-orchestrator runs it.
  • Does not replace plan-orchestrator. They handle different stages of the same pipeline.
For a small internal tool: SWR is probably not needed. If the scope is personal-use, low-risk, and easy to review, use the fast lane and reserve SWR for workflows involving money, customers, durable state, or broad blast radius.

optional memory   Lives at ~/keel/tools/gbrain/ (cloned from garrytan/gbrain, public, third-party). Installed only when you pass --with gbrain. A local-first memory and retrieval layer for agents.

What it is

An independent, third-party tool that gives agents a local memory database with stdio MCP integration. Keel documents it as an optional integration, not a required dependency.

./install.sh --with gbrain
# clones garrytan/gbrain
# bun install && bun link
# health check: bun --version

Recommended topology

  • Local database; no public HTTP exposure on day one.
  • Local stdio MCP server.
  • Explicit API-key handling only if you enable embedding providers that require one.

Boundary

gbrain remains owned by its upstream repository. Keel's responsibility is only to declare an optional install path and document how it fits into a local workflow.

Why isn't this part of the main flow? Because Keel's core idea-to-ship loop is intentionally stateless across tools. Adding a shared memory layer is useful but optional — every load-bearing handoff still happens through a written contract, not a shared memory.

8. Two paths through the engine

⚡ Fast lane (compiler)

Use this for low-risk planning. The compiler emits a scaffold that humans review before PO executes it.

1. /office-hours (gstack)
   → design doc

2. /autoplan (gstack)
   → approved build brief

3. keel-compile compile --dry-run
   → markdown_playbook_v1 scaffold
   → PO list-items + doctor (automatic)

4. Human review
   → accept · revise · discard scaffold

5. keel-run supervise run --next
   → completed run branch

6. /ship + /land-and-deploy (gstack)
   → live in production

The deterministic compiler author is scaffold-only today. Non-dry-run stub output is blocked unless an operator explicitly passes --allow-stub-output.

🛡 High-stakes lane (SWR)

Use this when the human owner judges the stakes high: security, money, irreversible writes, compliance, or customer-facing claims.

1. /office-hours (gstack)
   → design doc

2. /autoplan (gstack)
   → approved build brief

3. keel-swr supervisor init-session
   keel-swr run (5 staged calls)
   → reviewed markdown_playbook_v1
   → PO list-items + doctor

4. Human review
   → accept · revise · discard playbook

5. keel-run supervise run --next
   → completed run branch

6. /ship + /land-and-deploy (gstack)

Slower and more expensive. Worth it when the cost of a bad plan is higher than the cost of a careful review process.

Both lanes target the same contract, not the same trust level. Plan-orchestrator only executes a file that validates as markdown_playbook_v1, and both lanes run PO list-items + doctor before execution. Human review is mandatory in either lane.

10. The dual-audit pattern

The most load-bearing trick in plan-orchestrator: after verification, PO freezes a packet of evidence and gives the exact same packet to Codex and Claude. They review independently, and their findings get canonicalized by hash so duplicates collapse cleanly. Two witnesses, one truth.

Click each role to learn what is in the packet, why it is frozen, and how findings are canonicalized.
Why dual audit? One reviewer can be subtly biased by its own implementation. Two independent reviewers, looking at exactly the same evidence and canonicalizing findings by content hash, catch the bias. The kernel does not interpret findings — it only deterministically merges them.

11. Walkthrough: a feature, phase by phase

One concrete example, traced end to end. Click each phase to expand commands and artifacts.

phase −1 One-time bootstrap (do this once per machine) 
git clone https://github.com/AysajanE/keel.git ~/keel
cd ~/keel
./install.sh                # full install (private compiler may need access)
# or: ./install.sh --public-only --skip-tools   # works without compiler access
source ~/keel/keel.env

# Sanity-check the wrappers
keel-compile --help
keel-run --help
keel-doctor --help
keel-swr

Result on disk:

~/keel/tools/compiler/                  # cloned at manifest commit
~/keel/tools/plan-orchestrator/         # cloned at manifest commit
~/keel/tools/staged-workflow-runner/    # cloned at manifest commit
~/keel/keel.env                         # exports KEEL_ROOT, KEEL_PO_ROOT, …
~/keel/tools.lock                       # records resolved SHAs
phase 0 Idea → design doc

In Claude Code, you typed:

/office-hours

You said: "I want to build a small feature that turns a reviewed product idea into a concrete implementation plan."

Six rounds of forcing questions later, you had a design doc:

~/.gstack/projects/my-product/
  local-my-product-feature-design-YYYYMMDD-HHMMSS.md

Human gate:

  • You confirm the design doc reflects what you actually want to build.
phase 1 Design → reviewed plan → approved brief

Promote the design doc into the target repo, then review it:

# Copy the gstack design doc into the product repo
export PRODUCT_REPO="$HOME/workspaces/my-product"
cp ~/.gstack/projects/my-product/local-my-product-feature-design-YYYYMMDD-HHMMSS.md \
   "$PRODUCT_REPO/docs/gstack/my-feature-office-hours.md"

# In Claude Code, run the review gauntlet
/autoplan docs/gstack/my-feature-office-hours.md

This walks through CEO review → design review → eng review → DX review, with you making decisions at each step. Output:

docs/gstack/my-feature-autoplan.md
docs/briefs/my-feature.approved-brief.md

Human gate:

  • You confirm the approved build brief is what you want to execute.
phase 2A Brief → playbook scaffold (fast lane)

Run the compiler through its Keel wrapper:

source ~/keel/keel.env
export PRODUCT_REPO="$HOME/workspaces/my-product"

keel-compile compile \
  --repo-root "$PRODUCT_REPO" \
  --design   "$PRODUCT_REPO/docs/gstack/my-feature-office-hours.md" \
  --autoplan "$PRODUCT_REPO/docs/gstack/my-feature-autoplan.md" \
  --approved-brief "$PRODUCT_REPO/docs/briefs/my-feature.approved-brief.md" \
  --out      "$PRODUCT_REPO/docs/playbooks/my-feature.playbook.md" \
  --plan-orchestrator-root "$KEEL_PO_ROOT" \
  --human-approved-by "$USER" \
  --dry-run

The deterministic row author is scaffold-only today. To write stub output without dry-run, an operator must pass --allow-stub-output, and the compiler forces generated rows through signoff gates.

Outputs three artifacts plus a contract verification call:

docs/playbooks/my-feature.playbook.md           # candidate playbook
docs/playbooks/my-feature.playbook.meta.json    # compiler provenance
docs/playbooks/my-feature.validation.json       # validation report

# Then automatically:
keel-run list-items --playbook docs/playbooks/my-feature.playbook.md
keel-doctor          --playbook docs/playbooks/my-feature.playbook.md

If the PO root is intentionally unavailable, the operator must pass --skip-po-verify "<reason>" and record why the contract check was deferred.

Human gate:

  • Read the playbook end to end and confirm it is what PO should execute.
  • Validation status is pass and PO contract verification passed (or has a recorded skip reason).
phase 2B Brief → playbook (high-stakes lane, optional)

For low-risk work, skip this and use the compiler scaffold path. For high-stakes work:

# 1. Initialize an SWR session
keel-swr supervisor init-session \
  --root "$PRODUCT_REPO" \
  --clarified-task-brief docs/briefs/<slug>.approved-brief.md \
  --summary "Emit a reviewed markdown_playbook_v1 for <slug> v1."

# 2. Stage + examine + dry-run the playbook-emission scaffold
keel-swr supervisor stage-scaffold     ...
keel-swr supervisor examine-scaffold   ...
keel-swr supervisor dry-run-scaffold   ...

# 3. Run the 5 staged calls (each with multi-agent review between)
keel-swr run \
  --root "$PRODUCT_REPO" \
  --workflow-file automation/task_packs/gstack_design_to_po_playbook/workflows/gstack_design_to_po_playbook.workflow.json \
  --primary-job-input docs/briefs/<slug>.approved-brief.md \
  --skip-token-count --wait

# 4. Promote the terminal artifact
cp <SWR_RUN_DIR>/stages/05_final_markdown_playbook/response.final.md \
   docs/playbooks/<slug>.playbook.md

# 5. Validate (same as the fast lane)
keel-run list-items --playbook docs/playbooks/<slug>.playbook.md
keel-doctor         --playbook docs/playbooks/<slug>.playbook.md --format json
phase 3 Playbook → executed code, one item at a time

Run plan-orchestrator under supervision:

source ~/keel/keel.env
cd "$PRODUCT_REPO"

# Acknowledge any ambient agent configs once after reviewing them
export PLAN_ORCHESTRATOR_CLEAN_ENV_CONFIRMED=1

keel-run supervise run \
  --playbook docs/playbooks/my-feature.playbook.md \
  --next

For each item, plan-orchestrator:

  1. Creates a fresh git worktree at .local/automation/plan_orchestrator/worktrees/<RUN_ID>/item-01-attempt-1/
  2. Executes the item (Codex implements the action)
  3. Runs the verification command(s) inside the worktree
  4. Freezes an audit packet (candidate patch · verification report · frozen playbook snapshot · normalized plan · execution report)
  5. Asks Codex to audit the frozen packet
  6. Asks Claude to audit the same frozen packet
  7. Deterministically merges findings, hashing duplicates
  8. Triages: passed · awaiting_human_gate · blocked_external · escalated

If the item lands passed, plan-orchestrator fast-forwards the run branch orchestrator/run/<RUN_ID> and moves to the next item.

You can watch live:

keel-run supervise status --run-id <RUN_ID> --format json --exit-code
phase 4 Manual gates — you, in a human-held terminal

When plan-orchestrator reaches awaiting_human_gate, it stops. You read:

  • manual_gate.json
  • candidate patch
  • verification report
  • Codex audit report
  • Claude audit report
  • triage report

Then in your own terminal (no agent), you record the decision:

keel-run mark-manual-gate \
  --run-id <RUN_ID> \
  --item 02 \
  --decision approved \
  --by "$USER" \
  --note "Reviewed the gate packet and approve continuation." \
  --evidence-path docs/reviews/my-feature-02-signoff.md \
  --approval-token-file /secure/local/path/manual-gate-token.txt

Then resume:

keel-run supervise resume --run-id <RUN_ID>
Critical rule: no agent, no wrapper, no skill, no compiler ever calls mark-manual-gate. You do. This is the single unbreakable boundary of the system.
phase 5 External evidence — when the item needs proof from outside the repo

Some items need evidence plan-orchestrator cannot fetch on its own (e.g., "verified the Oura webhook works for my actual account"). When it reaches blocked_external, you supply local evidence:

# Put proof files in a local directory
mkdir -p /tmp/oura-evidence
# (drop screenshots, logs, etc.)

keel-run supervise resume \
  --run-id <RUN_ID> \
  --external-evidence-dir /tmp/oura-evidence

plan-orchestrator copies the directory into the item's evidence folder and resumes.

External evidence is local-file only. No agent fabricates it, no script downloads it. You hand it over physically.
phase 6 Passed branch → PR → deploy → retrospective

When all items reach passed, plan-orchestrator's work is done. The completed code lives on branch orchestrator/run/<RUN_ID>.

Hand off to gstack — but never directly:

# 1. Create a normal ship branch from the orchestrator run branch
git fetch origin
git checkout -b ship/my-feature orchestrator/run/<RUN_ID>

# 2. Let gstack prepare the PR
/ship                  # tests · review · version bump · CHANGELOG · open PR

# 3. Let gstack merge and deploy after the PR exists
/land-and-deploy

# 4. (For web apps) post-deploy monitoring
/canary

# 5. Documentation sync
/document-release

# 6. Capture learnings
/retro
/learn
Why a separate ship branch? orchestrator/run/<RUN_ID> is internal kernel state. gstack /ship is a release workflow that does its own merging, version bumping, and PR opening. Keeping them separate means you never confuse "the executor's working state" with "the thing we're proposing to ship."

12. What happens after plan-orchestrator finishes

Plan-orchestrator's last act is to report one of four terminal states. The next move is yours — but the playbook tells you what to do.

✓ passed

All items completed cleanly. orchestrator/run/<RUN_ID> is fast-forwarded with the work.

Next: git checkout -b ship/<slug> orchestrator/run/<RUN_ID>, then gstack /ship.

⏸ awaiting_human_gate

An item declared manual_gate and is waiting for your signoff.

Next: read the gate packet, then mark-manual-gate from your terminal, then supervise resume.

⏸ blocked_external

An item needs evidence that lives outside the repo.

Next: assemble the evidence directory, then supervise resume --external-evidence-dir.

⚠ escalated

Critical unresolved issue, infrastructure failure, or rejected manual gate. No automatic recovery path.

Next: read the audit reports and triage report, decide whether to revise the playbook, fix manually, or abandon the run.

The system is designed so "stop and ask" is always available. No matter where plan-orchestrator ends up, you can keel-run supervise status, keel-doctor, or inspect the worktree manually. Nothing is ever silently abandoned.

13. Trust boundaries that cannot be crossed

The system has carefully separated authorities. Crossing any of these lines breaks the safety properties that make the whole thing trustworthy.

🚫 No agent approves manual gates

Only you, from a human-held terminal, can call keel-run mark-manual-gate. Compilers, supervisors, skills, wrappers — none of them. Optionally protect this further with the env var PLAN_ORCHESTRATOR_MANUAL_GATE_TOKEN_SHA256.

🚫 No fabricated external evidence

When plan-orchestrator says blocked_external, evidence must come from a local directory you physically supply. No agent invents proof.

🚫 No broad write roots

allowed_write_roots must be narrow (1–3 entries, repo-relative). Never ., src, tests. Never .local, .git, .codex, .claude, .mcp.json, secrets, .env*, ops/config.

🚫 No skipping verification

Behavioral items (requires_red_green: true) must have at least one required_verification_commands entry that would fail without the change. No exceptions.

🚫 No merging supervisors

Plan-orchestrator's supervisor and SWR's supervisor stay separate. Merging them adds failure surface without improving anything.

🚫 No shipping the run branch directly

orchestrator/run/<RUN_ID> is internal state. Always create a sibling ship/<slug> branch and let gstack /ship drive the PR.

🚫 No vendoring tools into Keel

Tools must remain in their canonical repos. Keel only declares them in tools.manifest.yaml. tools/ is an ignored install target, never a tracked source tree.

🚫 No undeclared health checks

scripts/manifest.py restricts health checks to a single-line regex allowlist. Expanding the allowlist is reviewed with the same care as shell-script changes.

🚫 No reserved-column authoring

Three columns must NOT be authored: change_profile, execution_mode, host_commands. PO derives them at normalization time. If you populate them, normalization fails.

14. "Which tool should I use?" decision guide

Q: I have an idea. Where do I start?

→ gstack /office-hours

Always. Even for tiny features. The forcing questions surface what you do not yet know.

Q: I have an approved design. How do I get a playbook?

Is the work high-stakes (security, money, irreversible state, broad blast radius)?

No (most cases) → compiler. Fast, deterministic.
Yes → SWR. Five staged calls, multi-agent reviewed between each.

Q: My playbook is ready. How do I execute it?

→ keel-run supervise run --playbook ... --next

Always. The compiler and SWR never execute. Plan-orchestrator is the only executor.

Q: I just want to validate a playbook without running it.

→ keel-doctor --playbook <path> --format json

Shortcut for keel-run doctor --playbook .... Read-only; sets the ambient-agent acknowledge env var automatically because no code executes.

Q: plan-orchestrator stopped at awaiting_human_gate. What now?

Open a fresh terminal that no agent controls. Read the gate packet. If you approve:

keel-run mark-manual-gate --decision approved --by "<you>" \
  ... --approval-token-file /secure/local/manual-gate-token.txt
keel-run supervise resume --run-id <RUN_ID>

Q: plan-orchestrator finished with passed. How do I ship?

→ git checkout -b ship/<slug> orchestrator/run/<RUN_ID> → gstack /ship → gstack /land-and-deploy

Always create a sibling ship branch. /ship never operates directly on the kernel's run branch.

Q: I want to add a memory layer for my agents.

→ ./install.sh --with gbrain

Optional. gbrain is a separate third-party tool. It is not required for any core idea-to-ship workflow.

16. Glossary

meta-repoA repository that ships only coordination code (wrappers, manifest, installer, docs). Keel is one. Tools live in their own canonical repos and are cloned in at install time.
cockpitWhere humans think and decide. gstack at the start, gstack at the end.
compilerFast deterministic translator from gstack design doc to markdown_playbook_v1. Lives at ~/keel/tools/compiler/.
SWRStaged-workflow-runner. Optional high-stakes playbook compiler with multi-agent staged review. Lives at ~/keel/tools/staged-workflow-runner/.
plan-orchestrator (PO)The execution kernel. Lives at ~/keel/tools/plan-orchestrator/.
kernelSynonym for "executor" / "execution engine". Specifically the part of plan-orchestrator that owns run_state.json and the state machine. Supervisor wrappers are explicitly not kernel.
gbrainOptional third-party memory layer (garrytan/gbrain). Installed via ./install.sh --with gbrain.
install-time contracttools.manifest.yaml. Declares which tool versions Keel trusts, with strict allowlists and a release-gate flag.
runtime contractmarkdown_playbook_v1. The only artifact the compiler/SWR may emit, the only artifact PO consumes.
IRIntermediate representation. gstack_plan_ir_v1.json. What the parser hands to the row author.
candidate rowspo_candidate_rows_v1.json. Pre-validation rows the (future) LLM author emits.
validation reportcompiler_validation_report_v1.json. Fail-closed verdict on safety, paths, write roots, gates, verification.
supervisorThe additive layer around either plan-orchestrator (heartbeats, recovery) or SWR (review loop). Never authoritative for state.
worktreeAn isolated git working tree for one item attempt at .local/automation/plan_orchestrator/worktrees/<RUN_ID>/item-<ITEM_ID>-attempt-<N>/.
audit packetA frozen bundle of evidence (patch + verification report + snapshots) given to Codex and Claude simultaneously for independent review.
terminal stateOne of passed, awaiting_human_gate, blocked_external, escalated. Plan-orchestrator stops at every terminal.
manual gateA human-only stop. Only you can resolve it. keel-run mark-manual-gate is the kernel's privileged write boundary.
run branchorchestrator/run/<RUN_ID>. The internal branch plan-orchestrator fast-forwards as items pass. Never ship directly from here.
ship branchship/<slug>. The sibling branch you create from the run branch for gstack /ship to operate on.
keel.envGenerated, gitignored. Exports KEEL_ROOT, KEEL_COMPILER_ROOT, KEEL_PO_ROOT, KEEL_SWR_ROOT, KEEL_GSTACK_HOME, KEEL_BIN, KEEL_PYTHON, KEEL_VERSION.
tools.lockGenerated, gitignored. Records the resolved git SHA each tool was cloned at, so you can audit what is actually running on this machine.
